# Cobolens — UX & Visualization Design Study*Research brief for a free, open-source, local desktop app that points at a COBOL codebase and produces an interactive dependency map, diagrams, section summaries, and AI chat. Goal: a "wicked," beautiful, Sourcetrail-class interface built on the best available design and visualization principles.***Date:** 2026-06-29---## Executive SummaryThe single most useful precedent for this product is **Sourcetrail** — the beloved, now-archived open-source source explorer (16.5k GitHub stars). Its core insight is worth copying wholesale: pair an **interactive, symbol-centric graph** with a **synchronized code panel**, and drive everything from a **fast fuzzy search**. The graph never tries to show the whole codebase at once — it centers on the *selected symbol* and shows only its incoming/outgoing relationships, so the user explores *incrementally* without ever facing a "hairball." That focus-and-expand model is the heart of why people loved it, and it maps perfectly onto a COBOL program where you want to start at one program/paragraph/copybook and trace dependencies outward.For the build, the recommended shape is:- **Layout:** A three-pane, IDE-style dark interface — **Graph (center, dominant) + Code (right, synchronized) + AI Chat / Summaries (right rail or bottom dock)**, with a left navigator (search, file/program tree, history breadcrumbs). This is Sourcetrail's proven graph+code split, extended with the chat/summary surface the founder wants.- **Visualization library:** **Sigma.js + graphology** as the primary graph renderer (WebGL, scales to tens of thousands of nodes, strong TypeScript typing, worker-based layout), with **ELK or dagre** for hierarchical/dependency layout and **Mermaid** for static, exportable section diagrams (call flow, copybook hierarchy). If the graphs stay reliably under a few thousand visible nodes after clustering, **Cytoscape.js** is a strong alternative because its built-in graph algorithms (paths, centrality) come "for free."- **The non-negotiable UX rule:** never render the full graph. Use **progressive disclosure** — overview ? focus on a symbol ? expand neighbors ? detail on demand — exactly as Sourcetrail did, and as graph-viz UX research (Cambridge Intelligence, IBM's "overview first, zoom and filter, details on demand") prescribes.The rest of this document supports those recommendations with detailed findings.---## 1. Sourcetrail — The North Star**Repo (archived):** https://github.com/CoatiSoftware/Sourcetrail**Documentation:** https://github.com/CoatiSoftware/Sourcetrail/blob/master/DOCUMENTATION.md**Deep walkthrough w/ screenshots:** https://www.cppstories.com/2017/10/sourcetrail/**Wikipedia:** https://en.wikipedia.org/wiki/Sourcetrail### What it wasA free, cross-platform, **offline** source explorer (C/C++, Java, Python) that helped developers "get productive on unfamiliar source code." It was built by Coati Software (Salzburg, Austria), open-sourced under GPL-3.0 in 2019, and archived on 14 Dec 2021 (final release 2021.4.19). It earned **16.5k stars and 1.7k forks** — an unusually beloved developer tool. Source: https://github.com/CoatiSoftware/Sourcetrail### The interaction model (copy this)Sourcetrail was built on **three core concepts** (per its own docs and the C++ Stories walkthrough at https://www.cppstories.com/2017/10/sourcetrail/):1. **Graph** — an interactive, *dynamic* graph **built from the currently selected symbol**. It can center on a class, file, macro, function, field, or even a template parameter, and shows that symbol in an "active" state with **all incoming and outgoing dependencies** to related symbols. Crucially, it is *not* a static whole-program diagram — it regenerates around whatever you click.2. **Code** — a synchronized source panel showing the code around the selected symbol, including **snippet views** at every relevant usage site (so clicking a field highlights, in the code, every place it's used).3. **Search & Navigation** — a powerful **fuzzy-matching** search box (you can skip characters while typing) with autocompletion, plus Home/overview reset and history navigation.### The layout (copy this)**Graph on one side, source code on the opposite side, simultaneously** — a two-window split where selecting a node in the graph drives the code panel and vice versa. A search/navigation toolbar sits on top with Home (reset to overview), fuzzy search, and back/forward history. Reference screenshot lives in the repo: https://github.com/CoatiSoftware/Sourcetrail/blob/master/docs/readme/user_interface.png### Why people loved it (lessons to copy)From the C++ Stories review (https://www.cppstories.com/2017/10/sourcetrail/) and the repo:- **Instant payoff per click.** "In a microsecond I get so much valuable information" — clicking a node shows relations (members, params, references, specializations) on one side and readable code on the other.- **Beautiful, animated graphs.** "Really awesome looking" diagrams with smooth animations — the aesthetic mattered and drove delight ("fun to see... play with the architecture").- **Meaningful color coding.** Blue = variables/fields, yellow = functions/methods, grey = classes; edges colored by relationship type (e.g., usage, override). Consistent semantic color let users read structure at a glance.- **Powerful fuzzy search** as the primary entry point — you rarely scroll; you type.- **Incremental exploration.** You start from one symbol and expand outward (callers, callees, base/derived) *without losing context*. Call graphs show both caller and callee sides.- **Graceful with broken/huge code.** It indexed even projects with compile errors and handled a 2-million-line codebase. (Highly relevant for messy legacy COBOL.)- **Offline, local, free.** Exactly the constitution of this product.### The weak points (lessons to *avoid*)- **Tool-switching friction.** Living outside the IDE meant context-switching; the reviewer wanted the view *next to* the code. ? For Cobolens, keep graph + code + chat **in one window** so there is no switch.- **Indexing setup pain.** Required a Clang compilation database; setup could be hard despite plugins. ? Make first-run indexing **one-click and forgiving** (COBOL parsing should not demand a build).### Why it died (a cautionary note, not a UX lesson)The team discontinued it not because of UX but because of **maintenance burden**: a complex cross-platform C++/Qt architecture, founders taking full-time jobs and relocating, and the difficulty of keeping multi-language indexers and build dependencies current. Sources: https://en.wikipedia.org/wiki/Sourcetrail and the authors' farewell post (archived) https://web.archive.org/web/20211115131149/https://www.sourcetrail.com/blog/discontinue_sourcetrail/. **Implication for Cobolens:** scope to *one language* (COBOL), keep the stack maintainable (web tech in a desktop shell rather than bespoke C++/Qt graph rendering), and lean on maintained OSS libraries.### Living forks worth watchingThe community has not let it die. Notable forks: **NumbatUI** (Quarkslab) https://github.com/quarkslab/NumbatUI and the EOL/continuation discussion https://github.com/CoatiSoftware/Sourcetrail/issues/1214. Useful to study their direction and pitfalls.---## 2. "Understand a Codebase Visually" Tools — What's Good and Bad### CodeSee**Site:** https://www.codesee.io/ · **Learning center:** https://www.codesee.io/learning-center/code-visualization- **Good:** Auto-generated **code maps** that give an instant "mental model" of cross-repo/service dependencies; strong **onboarding** value (cited 10 hrs/month saved on review for one team); a clever **PR-review overlay** that maps changed files onto the architecture so a reviewer sees blast radius before merge; "intuitive grouping" of code by logic/function; review **progress tracking** ("mark as reviewed"). Source: https://www.codesee.io/learning-center/code-visualization- **Bad:** Cloud/SaaS-oriented (opaque pricing, requires contact for quotes); product direction uncertain after GitKraken acquisition; repetitive/marketing-heavy docs. Source: https://toolradar.com/tools/codesee- **Lesson for us:** Steal the **"map changes onto the architecture"** and **onboarding-first** framing. Avoid the cloud dependency — our app is **local**, which is itself a differentiator for security-sensitive COBOL shops (banks, insurers, government).### Sourcegraph (+ Cody)**Site:** https://sourcegraph.com/ · **Cody docs:** https://sourcegraph.com/docs/cody- **Good:** Best-in-class **code intelligence** — precise, symbol-level **go-to-definition** and **find-all-references** across repos (not text search), and **dependency-graph understanding**. Cody layers **RAG-based AI chat** on top of that index (local file ? local repo ? remote repo context), which makes its answers far more accurate on large monorepos than file-window-only assistants. Sources: https://www.vantagelabsai.com/tools/sourcegraph-cody and https://www.augmentcode.com/tools/sourcegraph-cody-vs-qodo- **Bad:** Daily-coding UX is "less polished than Cursor / Copilot"; it's primarily search/chat, not a *spatial* visualization (no beautiful graph). Source: https://devtoolsreview.com/reviews/cody-review/- **Lesson for us:** The winning AI pattern is **index-grounded chat** — build a symbol index of the COBOL (programs, paragraphs, copybooks, CALLs, data items) and feed the AI **precise context** from it. The chat should be able to *drive the graph* ("show me everything that calls WS-CUSTOMER-RECORD") and the graph should be able to *seed the chat* (click a node ? "explain this section").### Other models, briefly- **Source Insight / SciTools Understand / CppDepend** (per https://www.cppstories.com/2017/10/sourcetrail/): mature, dense, "Swiss-army-knife" reference trees, call trees, class diagrams — powerful but visually dated and heavy. Good feature checklists, bad aesthetic. We want their *capability*, Sourcetrail's *delight*.---## 3. Graph Visualization Libraries — Tradeoffs at ScaleThe authoritative 2026 comparison used below: **PkgPulse Decision Guide** https://www.pkgpulse.com/guides/cytoscape-vs-vis-network-vs-sigma-graph-visualization-2026, plus practitioner data from Cylynx (https://www.cylynx.io/blog/a-comparison-of-javascript-graph-network-visualisation-libraries/) and Stephen Weber's large-graph survey (https://weber-stephen.medium.com/the-best-libraries-and-methods-to-render-large-network-graphs-on-the-web-d122ece2f4dc).### The headline scaling ruleRendering technology sets the ceiling (Cylynx / Weber):- **SVG** (e.g., D3 default): workable to **~2k nodes / ~2k edges**, then degrades.- **Canvas** (Cytoscape.js, vis-network): degrades around **~3k–5k nodes**.- **WebGL** (Sigma.js): smooth into **tens of thousands**; Sigma renders **100k+ edges** with default styles; 60 FPS pan/zoom at ~2.3k nodes / 7k edges, dropping below 30 FPS only past **~20k nodes / 56k edges**. Sources: https://www.sigmajs.org/ and the Sigma large-graph issue https://github.com/jacomyal/sigma.js/issues/239> **Important caveat (PkgPulse):** these are *directional*. Labels, edge density, hit-testing, and layout cost usually dominate the real experience. **Benchmark with a representative COBOL graph** (realistic node/edge counts, labels on, your layout) on target hardware before committing.### Library-by-library| Library | Renderer | Best for | Strengths | Watch-outs ||---|---|---|---|---|| **Sigma.js + graphology** | WebGL | **Large interactive graphs** (our likely primary) | Scales furthest; graphology is a strong, **strongly-typed** (`Graph<Node, Edge>`) data layer with a deep ecosystem — ForceAtlas2 (with **WebWorker mode** so layout doesn't block the UI), Louvain community detection, Dijkstra/A*, centrality; official `@react-sigma` bindings with event-driven reactivity. | Renderer only — algorithms/labels/custom shapes are assembly work; no built-in animations; custom node shapes mean WebGL/shader effort. Sources: https://www.pkgpulse.com/guides/cytoscape-vs-vis-network-vs-sigma-graph-visualization-2026 || **Cytoscape.js** | Canvas | **Graph-as-analysis-object** (small/medium) | Batteries-included: shortest path, betweenness centrality, BFS/DFS, Markov clustering, many layouts (cose, breadthfirst, concentric, dagre ext.), rich styling, animations — all in one API. | COSE/force layout runs **synchronously on the main thread** and can freeze interaction on graphs above a few thousand nodes; canvas ceiling ~5k. || **vis-network** | Canvas | **User-editable, physics network diagrams** | Easiest path to drag/drop, physics, **built-in clustering** (`cluster`, `clusterByConnection`, open-on-click), dynamic DataSet updates, events. Great for an editable/exploratory diagram. | Canvas ceiling; physics needs stabilization (then freeze with `physics:false`); no native React binding (use a ref). || **React Flow** | SVG/DOM | **Node-based editor UIs / workflow builders** | Beautiful, ergonomic React-native node UIs; great for custom node cards (e.g., a program node with metrics/badges); pairs with ELK/dagre for layout. | "May lack high-performance graph *exploration* features"; SVG/DOM ceiling makes it weak for thousands of dense nodes — better for curated subgraphs than the full map. Source: https://reactflow.dev/learn/layouting/layouting || **D3** | SVG (Canvas possible) | **Bespoke custom visuals** | Total control over appearance/behavior/animation. | You build everything; SVG is slow for many small elements. Use for custom *accents*, not the core engine. || **Mermaid** | SVG (static) | **Static, exportable section diagrams** | Text-defined diagrams; AI can generate them; trivial to embed/export. | Not interactive; large diagrams slow/unpredictable. (See §3.) || **ELK (elkjs) / dagre** | *Layout engines, not renderers* | **Hierarchical / DAG layout** for dependency trees | **dagre**: fast, simple, layered layout — first choice for a DAG. **ELK**: more algorithms (layered, mrtree, force), far more configurable, better maintained. Feed positions into Sigma/Cytoscape/React Flow. | ELK is a Java port (heavier, async). Sources: https://github.com/dagrejs/dagre and https://reactflow.dev/examples/layout/elkjs |### Recommendation for Cobolens- **Primary renderer: Sigma.js + graphology.** A large mainframe COBOL system (hundreds of programs, thousands of paragraphs/data items, dense CALL/PERFORM/copybook edges) will blow past Canvas limits the moment you reveal more than a focused subgraph. WebGL + graphology's worker-based layout and typed data layer is the safe, future-proof choice, and `@react-sigma` fits a React/Electron or Tauri shell.- **Layout: dagre for the common case** (dependency/call relationships are largely a DAG ? clean layered layout), **ELK** when you need richer options or nested clustering. Run ForceAtlas2 (graphology) **in a worker** for the organic "neighborhood" views.- **Static diagrams: Mermaid** for AI-generated and exportable section diagrams (a paragraph's PERFORM flow, copybook hierarchy, a program's CALL tree).- **Custom node cards: optionally React Flow** for curated, "card-rich" subviews (e.g., a single program's internals with badges/metrics) where node count is small and beauty/detail per node matters.- **If post-clustering the visible graph reliably stays under ~2–3k nodes,** Cytoscape.js becomes attractive purely because its built-in algorithms (centrality to find "hub" programs, shortest path between two programs, community detection to auto-cluster) save real build time. A reasonable hybrid: **graphology for analysis, Sigma for rendering** — you get the algorithms without the canvas ceiling.---## 5. Concrete Design Principles for Cobolens### 5.1 Recommended overall layoutA **dark, IDE-grade, three-pane** workspace (Sourcetrail's split, extended for AI):```+---------------------------------------------------------------------------+¦  Top bar: fuzzy search ? • breadcrumb history (? ? ¦ Home) • view modes      ¦+-----------------------------------------------------------------------------¦¦ LEFT      ¦            CENTER (dominant)              ¦  RIGHT                ¦¦ navigator ¦            INTERACTIVE GRAPH              ¦  +------------------+ ¦¦           ¦   (Sigma/WebGL, focus + expand)          ¦  ¦ CODE (synced)    ¦ ¦¦ • search  ¦                                          ¦  ¦ snippet views    ¦ ¦¦ • program ¦      ? selected symbol (active)          ¦  ¦ at usage sites   ¦ ¦¦   tree    ¦     ? ¦ ?  callers / callees / data       ¦  +------------------¦ ¦¦ • filters ¦                                          ¦  ¦ SUMMARY + AI CHAT¦ ¦¦ • legend  ¦   minimap ? (bottom-right)                ¦  ¦ "Explain this    ¦ ¦¦           ¦                                          ¦  ¦  section" ? graph¦ ¦+-----------------------------------------------------------------------------+```- **Center = graph, and it's the star.** Largest pane. Symbol-centric, focus-and-expand (never the whole map).- **Right-top = synchronized code** with snippet views at every usage site (Sourcetrail's killer feature).- **Right-bottom = AI chat + section summaries**, bidirectionally linked to the graph (click node ? explain; ask question ? highlight nodes). This is the founder's chat surface and the Sourcegraph/Cody lesson applied.- **Left = navigator:** fuzzy search, program/copybook tree, type filters (program / paragraph / data item / copybook / CALL), and a **persistent color legend**.- **Top = search + history + view modes** (Graph / City overview / Static diagram export). Fuzzy search is the primary entry point.- Panes are **resizable and dockable**; let power users collapse the code or chat to give the graph full bleed.### 5.2 Progressive disclosure — the cardinal ruleFollow IBM's mantra and Cambridge Intelligence's anti-hairball guidance: **"Overview first, zoom and filter, then details on demand."** Sources: https://cambridge-intelligence.com/how-to-fix-hairballs/ and https://cambridge-intelligence.com/blog/designing-intuitive-data-experiences-with-graph-visualizations/- **Never render all nodes.** Hairballs occur when you visualize everything; meaningful patterns vanish. Start with a **focused view of ~20–50 relevant nodes**, then expand on interaction.- **Default to a clustered overview** (cluster by program / by copybook / by community via Louvain). Show **collapsed cluster nodes** sized by contents; **drill in on click** (vis-network's open-cluster pattern, or graphology communities).- **Expand outward from the selection** — click a node to reveal its direct neighbors (callers/callees/data references); double-click to go deeper. Mirror Obsidian's local graph and Sourcetrail's "center on selected symbol."- **Filter aggressively:** by node type, by edge type (CALL vs PERFORM vs data-use vs copybook-include), by depth (1–N hops), by metric threshold (only show fan-in > X).- **C4-style zoom levels** (from Structurizr's mental model): *System overview ? program clusters ? program internals (paragraphs) ? code.*### 5.3 Visual encoding — semantic color & node designCopy Sourcetrail's **consistent semantic color** and Obsidian's **size-by-importance**:- **Color by node type** (one stable, learnable palette): e.g., programs, paragraphs/sections, data items (WS variables), copybooks, external CALLs each get a distinct hue. Keep the legend always visible.- **Color edges by relationship** (CALL, PERFORM, READ/WRITE file, data reference, copybook include) — and **route edges to minimize overlap** (the lesson from Ghidra's disliked edge rendering).- **Size nodes by importance** (fan-in / fan-out / LOC) so hubs are obvious at a glance (Obsidian; CodeCharta's metric encoding).- **Active/selected state** is unmistakable; **dim non-neighbors** on hover/select (Sigma's `nodeReducer` highlight pattern) so the focus path pops.- **Smooth, purposeful animation** on expand/focus (Sourcetrail's delight factor) — but cheap enough to stay at 60 FPS.### 5.4 Dark mode & typography for a "sharp, fast, modern" feelFrom dark-mode UX guidance (https://blog.logrocket.com/ux-design/dark-mode-ui-design-best-practices-and-examples/, https://designcode.io/dark-interfaces/, https://www.halo-lab.com/blog/dark-ui-design-11-tips-for-dark-mode-design):- **Design dark mode as its own system — never an inverted light theme.** This is the #1 mistake.- **Avoid pure black and pure white.** Use a very dark neutral (e.g., `#0E1116`–`#1A1D23`) for the canvas and **off-white / light grey** text (pure white is harsh for long sessions). Use **elevation via subtle lighter surfaces** (panels, menus, popovers get progressively lighter) rather than heavy borders.- **Slightly reduce font weight** in dark mode (light-on-dark looks optically heavier — e.g., 350 vs 400).- **Typography:** a crisp UI sans (Inter / SF / system) for chrome; a **monospace** (JetBrains Mono / Fira Code / SF Mono) for the code panel, node labels, and identifiers — it signals "developer tool" and aids alignment. Body ~13–14px, line-height ~1.45–1.5.- **Restrained accent color** used only for the active symbol, primary actions, and the focus path — so the eye always knows what's selected. Reserve red/amber strictly for warnings (dead code, missing copybook), not decoration.- **Generous negative space + a clear grid**; the "sharp/fast" feel comes from restraint, consistent spacing, and snappy interactions, not decoration.### 5.5 Performance considerations- **Index once, cache aggressively** (Sourcetrail's model): parse the COBOL into a symbol DB; re-index only changed files. Indexing must be **forgiving** of dialect quirks and partial/broken code (legacy COBOL is messy).- **Pre-compute layout off the main thread** (graphology ForceAtlas2 worker mode; or precompute dagre/ELK positions at index time and store them) so opening a view is instant and pan/zoom never stutters.- **Level-of-detail:** hide labels and thin edges when zoomed out; reveal them on zoom-in (avoids label thrash and keeps FPS up).- **Cap visible nodes** with clustering + on-demand expansion rather than throwing 2000 nodes at WebGL at once — keeps interaction crisp *and* readable.- **Always-on minimap** for orientation in large graphs (IDA/Ghidra lesson: navigation is what makes dense graphs usable).- **Benchmark with a real, large COBOL graph** before locking the library choice (PkgPulse caveat).### 5.6 Onboarding & first-run- **One-click "Open a COBOL folder"** ? progress-tracked indexing ? **land on a clustered overview**, not an empty canvas. (Address Sourcetrail's setup-pain and Sourcegraph's "where do I start?" gap.)- **Ship a sample COBOL project** so users can explore in 10 seconds without their own codebase (Sourcetrail shipped example projects).- **A "Home"/overview reset** always available (Sourcetrail's Home button) so users can never get lost.- **Inline coaching:** tooltips, a one-time guided tour highlighting search ? click ? expand ? ask-AI, and an empty-state that *teaches the interaction model* ("Search for a program, then click to expand its callers").- **Let the AI be a guide:** seed the chat with prompts like "What does this program do?", "Show me everything that touches WS-CUSTOMER-RECORD", "Find dead paragraphs" — and have those answers **drive the graph**.---## Do This / Avoid This — UX Rules### ? Do- **Do** make the graph **symbol-centric and focus-and-expand** (Sourcetrail) — start at one node, grow outward.- **Do** keep **graph + code + AI chat in a single synchronized window** — clicking a node updates code *and* offers an AI explanation; AI answers highlight nodes.- **Do** make **fuzzy search the primary entry point**, with autocomplete and skip-character matching.- **Do** use **consistent semantic color** for node types and edge types, with an always-visible legend.- **Do** **size nodes by importance** (fan-in/fan-out/LOC) so hubs are obvious.- **Do** **cluster by default and drill down** (Louvain communities / by-program / by-copybook).- **Do** offer a **minimap, breadcrumb history, and a Home reset.**- **Do** **route edges to minimize overlap** and use **level-of-detail** (labels/edges fade out when zoomed out).- **Do** **render with WebGL (Sigma) and lay out in a worker** so it stays at 60 FPS.- **Do** **index forgivingly and cache** — handle broken/partial legacy COBOL gracefully.- **Do** **land users on a clustered overview** with a sample project available on first run.- **Do** design **dark mode as its own system**: near-black canvas, off-white text, elevation by surface lightness, monospace for code/identifiers, one restrained accent for the focus path.### ? Avoid- **Avoid** ever rendering the **entire graph** — hairballs hide meaning (Obsidian global graph, Ghidra). Progressive disclosure always.- **Avoid** **overlapping, ambiguous edges** — the single most-cited reason Ghidra's graph view is disliked.- **Avoid** **inverting a light theme** into "dark mode," **pure black** backgrounds, and **pure white** body text.- **Avoid** **main-thread layout** on big graphs (Cytoscape COSE) — it freezes the UI.- **Avoid** **tool-switching friction** — don't make users leave the app to read code (Sourcetrail's weak point).- **Avoid** **painful first-run setup** — no build configs or compilation databases required to index COBOL.- **Avoid** **Mermaid for the interactive map** — it's static, slow on large diagrams, and lays out unpredictably; reserve it for exportable section diagrams.- **Avoid** **decorative color** — color must encode meaning (type/relationship/state), not just look pretty.- **Avoid** **citing universal node-count limits** in planning — benchmark your own COBOL graph shape on target hardware.- **Avoid** **dropping users on
